1. Introduction

 

The availability of tools to capture, display and interpret packets traversing a network has proven extremely useful in debugging network problems. The ability to capture packets and store them for later analysis allows one to de-couple the tasks of collecting information about a network problem and analyzing that information. The jNetStream project provides programs with ability to capture packets and store them in a file, and can further interpret the packets stored in capture files. This paper describes the file format that the jNetStream programs use to store captured packets. This paper was written so that others may write programs to interpret the capture files generated by jNetStream programs, or create capture files that can be interpreted by jNetStream programs.

 

 

2. File Format

 

The packet capture file is an array of octets structured as follows:

 

     +-------------------------+

     |                         |

     |         Block           |

     |        Number 1         |

     +-------------------------+

     .                         .        

     .                         .        

     .                         .        

     +-------------------------+

     |                         |

     |         Block           |

     |        Number N         |

     +-------------------------+

 

The capture file consists of a one or more blocks of octet data. Each Block can be up to 4 gigabytes in length and no more. The overall file can be any size by containing more than one block.

 

The Block length is typically much smaller then maximum possible of 4 gigabytes and can be specified by tool or user at the time of file creation. The default length is 512 kilo bytes (512 * 1024 bytes).

 

Each Block is also an array of octets structured as follows:

 

     +-----------------------------+

     |                             |

     |   Block Record header     |

     |          Number 1           |

   +-----------------------------+

   | +-------------------------+ |

   | |                         | |

   | |        Record           | |

   | |        Number 2         | |

   | +-------------------------+ |

   . .                         . .      

   ~ .                         . ~      

   . .                         . .      

   | +-------------------------+ |

   | |                         | |

   | |        Record           | |

   | |        Number N         | |

   | +-------------------------+ |

   +-----------------------------+

   

 

 

The first record Number 1 is always a Block Record defined in section 4.1. The Block Record contains general information about the Block.

 

Record Number 2 is always preceded by a Block Record, and it must appear after the Block Records header, or more exactly at 25th byte offset into the octet array of the Block.

 

Record 3 through N, must appear after the last byte of the previous Record before it (n  1). This is the last by of the entire record, not just the header as specified by the Record Length field within the Record header. See section 3.3.

 

 

 

 

 

 

3. Record format

 

Each record is an array of octets with the following format:

 

      1       4        8       12      16      20      24      28      32

      -------------------------------------------------------------------

      | Type  |                                                         |

      |-----------------------------------------------------------------|

      |                         Record Length                           |

      |-----------------------------------------------------------------|

      |                                                                 |

      |-----------------------------------------------------------------|

      |                                                                 |

      |-----------------------------------------------------------------|

      |                                                                 |

      .                                                                 .

      ~                           Record Data                           ~

      .                                                                 .

      +                                                      +- - - - - +

      |                                                      |   Pad    |

      |-----------------------------------------------------------------|

 

3.1 Record header

 

Each Record has a fixed-length, 24-octet header, that defines two standard fields, Type and Record Length (see section 3.2 and 3.3) These two fields are present in every record of every type.

 

3.2 Record Type field

 

4-bit unsigned integer field. The possible range of values for this field is from 0 to 15 (0x0 to 0xF hexadecimal).

 

The values in this field, specify the type of record this is, and may have the following values:

 

Value     Record Type

-----     -----------

0x0       Packet Record

0x1       Block Record

0x2       Meta Record

0x3       Vendor Record

0xF       NoOp Record

 

Each record type is explained in section 4.

 

3.3 Record Length field

 

32-bit unsigned integer. Record length field contains the number of octets that make up the Record. The number includes the 24 octets that make up the Records header plus the Data and Pad fields.

 

3.4 Record Data field

 

Variable-length field containing zero or more octets that are Record Type specific.

 

   The Record Data field also contains the octets of the Pad field.

 

See section 3.5

 

3.5 Record Pad field

 

Variable-length field holding zero to seven octets that pads the Packet Record out to a convenient boundary for 32-bit and 64-bit systems. The alignment has to be on a 8 byte boundary.

 

Although strictly speaking, this is not a separate field, but in actuality is part of the Record Data field. It is described as a separate entity to highlight the fact that entire record length, the header and its data needs to be aligned to a 8 octet boundary.

 

The octet values that make up the pad may contain any value and do not need to be initialized.

 

Also note, there is no way to determine from just the record fields how many padding octets were applied to pad to a convenient boundary. Therefore it is up to each individual Record Type (see section 3.2) to determine how much of the Record Data field is really part of real data and how much is simply unused octets.

 

3.6 Reserved and Unused fields

 

All Reserved fields within the Record header must be reset to value of zero (0). It is illegal to leave these fields with any values other then zero (0).

 

Note, the Record Data field or Record Unused Data fields do not have to be initialized.

 

This general statement only applies to Reserved or Unused values within the Record header.

 

 

 

 

 

 

4. Types of Records and Their Formats

 

The NAP capture file format defines the following four record types that may or may not exist in the capture file block:

 

1)      Block Record

2)      Packet Record

3)      Meta Record

4)      Vendor Record

5)      NoOp Record

 

The special conditions are as follows.

 

1)      Every Block, as defined in section 2, must have one (1) and only one (1) Block Record. The Block Record must appear as the first record within each block.

2)      Certain Meta Records that are applicable and provide meta information about a Packet Record, must precede that Packet Record and may not appear after it.

3)      NoOp Record may appear anywhere within the block, except as the first record which must be a Block Record

 

4.1 Block Record Format

 

The Block Record is made up of a Block Record header and Data field. The Record Data field, is made up of other Records. The capture file is made up of one (1) or more of Block Records.

 

Only the Block Record and no other Records, may contain Records in their data field.

 

Each Block Record header defines certain general information about the entire Block Record (also called the Block.)

 

First, the Block Records Magic Number field contains an pattern, made up of several octets, which uniquely identifies the block and file type as NAP.

 

Secondly the Record Length field, standard in all records, defines the length in octets of the entire block. For capture files containing a single Block Record, this field is equal to file size. If more than one block is contained within the capture file, the sum of all the Block Record length fields within the file, equals the size of the entire capture file.

 

The length of a record, must be aligned and padded with 0 or more octets if needed to a 8 byte boundary.

 

 

The header format of a Block Record is as follows:

 

      1       4        8       12      16      20      24      28      32

      -------------------------------------------------------------------

      | Type  |                    Magic Number                         |

      |-----------------------------------------------------------------|

      |                           Record Length                         |

      |-----------------------------------------------------------------|

      |  Major Version | Minor Version |             Flags              |

      |-----------------------------------------------------------------|

      |                            Block Id                             |

      |-----------------------------------------------------------------|

      |                          Security Code                          |

      |-----------------------------------------------------------------|

      |                          Packet Count                           |

      |-----------------------------------------------------------------|

 

Type:

 

See section 3.2

 

Magic Number:

 

28-bit (3 octets + 4 bits) pattern used to identify the file as a NAP packet capture file. The identification pattern consists of the 4 hexadecimal octets:

 

2 AA 3B B4

 

Note that since the Type field is constant, the combined single 32-bit integer value of Type + Magic Number produces the following constant 4 octet hexadecimal pattern:

 

      12 AA 3B B4

 

Record Length:

 

See section 3.3

 

Major Version:

 

8-bit unsigned integer. Major version which defines binary compatibility of the block. This document describes version number 1.

 

 

 

 

Minor Version:

 

 

8-bit unsigned integer. Minor version which defines binary compatibility of the block.

 

Flags:

 

16-bit (2 octet) field containing certain information about the block. Each bit within the flag is interpreted separately.

 

Flag     Meaning

----     -------

0x0000   No flags

0x0001   Packet count is present

0x0010   Security Code present

0x0020   Scramble Records using Security Code

0x0040   Full Encryption

 

 

Block ID:

 

32-bit unsigned integer. A random number that uniquely, within reason, identifies this particular block.

 

See section 6.1

 

Packet Count:

 

32-bit unsigned integer. Number of Packet Records present within this block. This field is optional. The flag 0x0001 Packet Count must be set in the Block Record headers flags field. If the flag is not set, the value of the Packet Count field has to be initialized to integer value of zero (0).

 

Security Code:

 

32-bit unsigned integer. A simple Security Code that must be checked against user provided Code. This field is optional. The flag 0x0010 Block Security Code must be set in the Block Record headers flags field. If the flag is not set, then the value of the Security Code must be reset to zero (0).

 

See section 6.2

 

4.2 Packet Record Format

 

The Packet Record, contains the captured network packet data.

 

The Packet Record header has the following structure:

 

      1       4        8       12      16      20      24      28      32

      -------------------------------------------------------------------

      | Type  |  Capture System ID     |            DLT Type            |

      |-----------------------------------------------------------------|

      |                          Record Length                          |

      |-----------------------------------------------------------------|

      |                     Original Packet Length                      |

      |-----------------------------------------------------------------|

      |                     Included Packet Length                      |

      |-----------------------------------------------------------------|

      |                        Timestamp Seconds                        |

      |-----------------------------------------------------------------|

      |                      Timestamp Nanoseconds                      |

      |-----------------------------------------------------------------|

      |                                                                 |

      .                                                                 .

      ~                           Packet Data                           ~

      .                                                                 .

      .                                                                 .

      +- - - - - - - - - - - - - - - - - - - - - - - - - - - +- - - - - +

      |                 Unused Record Data                   |   Pad    |

      |-----------------------------------------------------------------|

 

 

 

Type:

 

See section 3.2

 

Capture System ID:

 

12-bit unsigned integer value representing a unique ID within this block, of a capture system that captured this packet. Multiple Capture Systems may be capturing packets, either from multiple interfaces on the same system, or remote systems and this field uniquely identifies that Capture System by this ID.

 

Each Capture System is described and assigned a unique ID within a Meta Record of Sub Type Capture System. The Meta Record describing the Capture System by its ID, must appear within the current Block, before a Packet Record with a non Unspecified Capture System ID is encountered.

 

Capture System ID    Description

-----------------    -----------

0                    Unspecified Capture System

1  4095             Dynamically allocated per

                     Block

 

 

 

Record Length:

 

See section 3.3

 

DLT type:

 

16-bit unsigned integer value that corresponds to the first protocol type of the data within the packet data. This usually corresponds to a data link layer type, that is, the first header of the packet data is contained within the Record Data field of this record.

 

Value          DLT Type

-----          --------

0              No Link-Layer encapsulations

1              Ethernet

2              Experimental 3Mb Ethernet

3              Amateur Radio AX.25

4              Proteon ProNET Token Ring

5              Chaos

6              IEEE 802.5 TokenRing

7              ARCNET, with BSD-style header

8              Serial Line IP

9              Point-to-Point Protocol

10             FDDI

11             LLC/SNAP encapsulated ATM

12             Raw IP

15             BSD/OS Serial Line IP

16             BSD/OS Point-to-Point Protocol

19             Linux Classical-IP over ATM

50             PPP over serial with HDLC

51             PPP over Ethernet (PPPoE)

99             Symantec Filewall

104            Cisco HDLC

105            IEEE 802.11 wireless

107            BSD/OS Frelay

108            OpenBSD loopback device

109            Encapsulated packets for IPSec

113            Linux cooked sockets

114            Apple LocalTalk

115            Acorn Econet

116            OpenBSD ipfilter

 

 

 

 

 

 

 

Original Packet Length:

 

32-bit unsigned integer representing the length in octets of the captured packet as received via a network.

 

 

Included Packet Length:

 

32-bit unsigned integer representing the length of the Packet Data field. This is the number of octets of the captured packet that are included in this packet record. If the received packet was truncated, either at capture or during file post processing, the Included Length field will be less than the Original Length field. The field may also be less then the Record Length reflecting the up to 8 octets of pad and possibly reduced or initially oversized Record Data field.

 

Timestamp Seconds:

 

32-bit unsigned integer representing the time, in seconds since January 1, 1970, the packet arrived.

 

Timestamp Nanoseconds:

 

32-bit unsigned integer representing nanosecond resolution of packet arrival time. Valid value for this field is between 0 and 999,999,999 inclusive.

 

Packet Data:

 

Variable-length field holding the packet that was captured, beginning with its DLT type or data link header. The DLT Type field of the file header can be used to determine how to decode that first header within this Packet Data. The length of the Packet Data field is given in the Included Length field.

 

Unused Record Data

 

Variable-length field holding unused octets. This is the difference between Record Length and Included Packet Length. Length of Packet Record may be more then is needed to store the captured packet for variety of reasons such as when packet data has been post processed and the Included Packet Length has decreased. Also it is possible that originally, the length of a record had been over allocated to accommodate expected modifications to the captured packet that would increase its size.

 

See section 3.5 for more information.

 

Pad

 

See section 3.5

See section 3.2

 

Record Length:

 

See section 3.3

4.4 Vendor Record format

 

Vendors can extend the file format by first acquiring a unique Vendor ID and then defining the format of the unused fields within the Vendor Record header.

 

Vendor records can be used to implement vendor specific functionality, that is unique to a particular Vendor only.

 

This should not be construed as a way of extending the general mechanism and format of the capture file. Any general, non vendor specific enhancements need to added in a standardized way so the new Block version can be allocated to indicate binary incompatibility of the enhancement.

 

Any implemented or unknown vendor in old version of the implementation program, can be safely ignored.

 

Vendors should not place block sensitive information within this type of record as it may not be recognized by all implementations reading the file. The Flags field should be set appropriately, and accurately, to ensure that a particular Block, containing other block with non vendor specific information, are not impacted by the processing of unknown vendor blocks.

 

 

 

Vendor Meta Record format has the following structure:

 

      1       4        8       12      16      20      24      28      32

      -------------------------------------------------------------------

      | Type  | Flags  |           Vendor ID                            |

      |-----------------------------------------------------------------|

      |                           Record Length                         |

      |-----------------------------------------------------------------|

      |                                                                 |

      |-----------------------------------------------------------------|

      |                                                                 |

      |-----------------------------------------------------------------|

      |                                                                 |

      |-----------------------------------------------------------------|

      |                                                                 |

      |-----------------------------------------------------------------|

      |                                                                 |

      .                                                                 .

      ~                           Record Data                           ~

      .                                                                 .

      +                                                      +- - - - - +

      |                                                      |   Pad    |

      |-----------------------------------------------------------------|

 

Type:

 

See section 3.2

 

Flags:

 

Flags which specify the treatment of this record

 

Value    Description

-----    -----------

0x0      Record can be ignored

0x1      Record can not be ignored

 

The flags help determine what action the implementing program must take if the Vendor ID is not recognized.

 

The value 0x0 Record can be ignored means that if the Vendor ID is not recognized, the record can be skipped over and ignored. The rest of the file can be interpreted normally, without the Vendor Record data.

 

The value 0x1 Record can not be ignored means that if the Vendor ID is not recognized, the record may not be skipped over and the implementation program must report and error and abort processing any more data within this block. This is the case when the Vendor record contains sensitive information that must be read before the remainder of the records can be processed, such as references and other possible pre-requisites. There can also be other reasons for not allowing the record to be skipped that are not discussed here.

 

Vendor ID:

 

24-bit unsigned integer that is a Unique ID assigned to a Vendor. It is illegal to assign or use unauthorized Vendor Ids.

 

Vendor Ids can be assigned by asking the Assignment Body for a new Vendor ID. See section {TODO: Assignment Authority}

See section 3.3

 

Record Data:

 

See section 3.4

See section 3.5

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

4.5 Meta Record Format

5. Meta Record Sub Types

 

The Meta Record carries additional Meta information about the capture file, Record Block or individual Records. Sub Type field of a Meta Record specifies exactly the type of information that is contained.

 

The Sub Type also determines the unassigned fields of the Meta Record and the structure of its Data field. Meta Records Data field contains Sub Type specific information.

 

5.1 Invalid Meta Record Sub Type

 

Sub Type = 0

 

Integer value of 0 in the Sub Type field of a Meta Record indicates that this is an invalid Record and possibly the file or just the block has been corrupted. This condition should be logged indicate that a corrupt record has been encountered.

 

5.2 Property Meta Record Sub Type

 

Sub Type = 1

 

Integer value of 1 in the Sub Type field of a Meta Record indicates that this Meta Record contains string properties.

 

The Property Meta Record header uses the unassigned values of the Meta Record header to describe the data in the Property Meta Record Data field.

 

 

The structure of a Property Meta Record is as follows:

 

      1       4        8       12      16      20      24      28      32

      -------------------------------------------------------------------

      | Type  |     Sub Type            |      Property Format          |

      |-----------------------------------------------------------------|

      |                          Record Length                          |

      |-----------------------------------------------------------------|

      |                     Number of Properties                        |

      |-----------------------------------------------------------------|

      |                          Line Capacity                          |

      |-----------------------------------------------------------------|

      |                             Reserved                            |

      |-----------------------------------------------------------------|

      |                             Reserved                            |

      |-----------------------------------------------------------------|

      |                                                                 |

      .                                                                 .

      ~                        Property Table                           ~

      .                                                                 .

      .                                                                 .

      +- - - - - - - - - - - - - - - - - - - - - - - - - - - +- - - - - +

      |                 Unused Record Data                   |   Pad    |

      |-----------------------------------------------------------------|

 

 

Type:

 

See section 3.2

 

Sub Type:

 

See section 5

 

 

Record Length:

 

See section 3.3

 

Property Format:

 

The format of each row in the Property Table field.

 

Format    Description

------    -----------

0         Attribute/Value pair (AVPair)

1  2^32  Unassigned

 

 

 

AVPair:

 

The AVPair format is structured as follows. The Property Table field contains Number of Properties count of rows. Each row has a single column containing an array of Unicode 8 characters for a total length of Line Capacity integer field.

 

Unused characters within the allocated table row, must be padded with empty/white space character.

 

The string line uses this suggested format:

 

Property = value

 

Or more formally:

 

PropertyLine := Attribute = Value PAD

Attribute := String

Value := String

PAD := WS*

String := Unicode*

WS :=  

;

Reserved:

 

See section 3.6

 

Property Table:

 

A table containing properties. The table size is allocated to be from the following formula and fields:

 

Number of Properties * Line Capacity

 

All text data is stored as Unicode characters which take up 2 octets of storage.

 

To index into the table and calculate the starting offset into the octet array of the Record Data field, using an unsigned integer variable Index:

 

Index * Line Capacity

 

Since all property line strings are constant in size, this is an easy calculation.

 

Storing values requires first to calculate the offset of the property line into the table and writing appropriate number of Unicode characters into the storage.

 

Care must be taken, not to go out bounds set by the Line Capacity field as this will result in overriding outside of the intended area. Such action may override next property or next Record header and destroy the integrity of the file.

 

Strings that are written, must be exactly the size of Line Capacity field. If the string that is to be written is shorter then the Line Capacity field, it must be padded with White Space.

 

It is illegal to leave shorter strings unpadded with White Space as those octets may contain 0 or other binary data that can not be interpreted as a Unicode character.

 

Unused Record Data:

 

See section 4.2

 

Note, that it is legal to expand the size of the Property Table field into the Unused Record Data area in some kind of post-processing of the file.

 

The headers Property Count field has to be updated appropriately to reflect the expanded table size.

 

It is illegal to change the Line Capacity field value once the header has been created. If the Line Capacity field must be changed, the implementation must replace the entire Record with a new Record with appropriate capacity.

 

Care must be taken not to go out of bounds and write data outside the Unused Record Data and Pad as that will override headers of the next Record and destroy integrity of the capture file.

 

 

Pad

 

See section 3.5

 

5.3 Capture System Meta Record format

 

A Capture System is an entity that has captured packet data and has stored it in this capture file within its Block. The Capture System may be used to uniquely identify the source of the packet data within this Block.

 

A single implementation program may be capturing data from multiple network interfaces and storing those captured packets within this block.

 

Multiple implementation programs, even on remote systems, may be capturing data from multiple interfaces and storing those captured packets into the same block within a single capture file.

 

Each such system is assigned a Unique ID, within this block and referenced as Capture System ID field within individual Packet Records in this Block.

 

This Unique Capture System ID is simply a zero-based index value into the tables in the Capture System Meta Record.

 

 

The format of the Capture System Meta Record is as follows:

6. Algorithms and Concepts

 

6.1 Block Records Block ID

 

Block ID number is calculated using one of the algorithms specified in the order in which they should be used.

 

The variables below A, B and Block ID are all 32-bit unsigned integers. Random Number generator is assumed to be generating a 32-bit unsigned integer values.

 

The Timestamp in seconds is assumed to be the number of seconds since January 1, 1970. The Timestamp in nanoseconds is assumed to be a 32-bit unsigned integer representing the fractional nanosecond part of a second. The Timestamp milliseconds is assumed to be a 32-bit unsigned integer representing the fractional millisecond part of a second.

 

The preferred algorithm is as follows, if the implementing system is capable of nanosecond time resolution:

 

   A = Current Timestamp in seconds

   B = Current Timestamp in nanoseconds

 

Block Id = A XOR B

 

The second preferred algorithm is as follows, if the implementing system is capable of millisecond time resolution:

 

   A = Current Timestamp in seconds

   B = Current Timestamp in milliseconds

 

   Block Id = A XOR B

 

 

The third preferred algorithm is as follows:

 

   A = Current Timestamp in seconds

   B = Number generated by Random Number generator

(Number generator must be properly seeded)

 

   Block ID = A XOR B

 

The forth preferred algorithm is as follows:

 

   A = Number generated by Random Number generator

(Number generator must be properly seeded)

 

   Block ID = A

 

 

The Block ID should be fairly unique, but not absolutely unique over long period of time. The ID has 1 in 4e10^9 chance of being in conflict with another ID previously generated. The number calculated by the above algorithms is sufficient enough for unique identification of different Record Blocks within a capture file, for most purposes.

 

If none of the above algorithms are possible on the implementing system, the implementing program, must calculate the value of block ID using a custom algorithm as described below.

 

It is noted, that super critical applications of the Block ID number should implement their own separate algorithm for calculating unique Block Ids. The custom calculated Block ID can be stored in the Block ID field. The uniqueness of the custom calculated Block ID, should be as good or better then using the above specified algorithms.

 

It is illegal to use for a Block ID, a constant, or a derivative of constant that will not produce unique Ids among different Record Blocks.

 

6.2 Block Records Security Code

 

The Security Code can be used as a simple password for the Block  where implementing program accessing the Block Records has to supply a challenge Code that is compared with the Block Records Security Code stored in the Block Record header.

 

If the supplied challenge code does not match the un-hashed Security Code calculated based on the Security Code field value, the request to read the remainder of the Records should be denied.

 

XOR is a bitwise operation that performs Exclusive OR against the left and right side of the expression.

 

The value within the field is stored as a hash of the actual code with the first 4 octets of the block record.

 

To compute the hashed Security code value, the formula is as follows:

 

1)   Convert first 4 octets of the block header into an unsigned integer which has a constant hexadecimal value of:

 

0x12aa3bb4

 

2)   Perform bitwise XOR operation on the Code value to be hashed with the constant from step 1.

 

Un-Hashed Code XOR 0x12aa3bb4

 

3) The result is stored in Security Code field.

 

To compute the true Security code value (un-hashed), the formula is as follows:

 

 

1)     Convert first 4 octets of the block header into an unsigned integer which has a constant hexadecimal value of:

 

0x12aa3bb4

 

2)     Read the hashed value form the Security Code field.

 

3)   Perform bitwise XOR operation on the hashed Code value with the constant from step 1.

 

Hashed Code XOR 0x12aa3bb4

 

4) The result is the original Security Code for the block. Do not store the non-hashed value in the Security Code field.

 

 

The value of the code is guessable by means of looking at the binary value in the capture file or guessed by trial and error. The purpose of this Security Code is to provide a simple mechanism to limit casual user from reading the file.

 

For security sensitive information the Encryption Option should be used to encrypt the Record Data Fields contained within this block. This option is much less efficient then simple Security Code

 

For maximum security, the entire file should be encrypted by external tools. The mechanism or such tools are not discussed in this paper.

 

 

 

7. Use cases

 

The following typical use cases are described which highlight some of the features that were designed into the NAP capture file format.

 

1)      All NAP capture files can always be concatenated together to form a single large capture file. The capture files Block Records are simply appended, one after the other, to the resultant file containing the output of the concatenation. This can be accomplished with any program or tool that can append bytes of one file to the other. Such a tool on Unix systems is cat.

2)      Any NAP capture file, can be split into multiple separate capture files. The resultant files are completely compatible with the NAP format. The split may only happen exactly on a Block boundary. This can be done with any program that can split a binary file exactly at a certain offset within the file. The default block size and thus its boundary is at 512 Kb.

3)      File Blocks can be password protected

4)      Record Data fields may be encrypted. Although the Record headers may not be encrypted.

5)      NAP capture file may store any number of user defined properties.

 

7.1 Concatenation of two capture files

 

Two NAP capture files exist. Both files are of NAP version 1. The first file is of size 1.0 Mb and the second file is 256 Kb in size.

 

The first file contains two Blocks of 512 Kb each, while the second file contains 1 Block of 256 Kb in length.

 

              File #1                           File #2

     +-------------------------+       +-------------------------+

     |                         |       |                         |

     |     Block size=512Kb    |       |    Block size=256Kb     |

     |        Number 1.1       |       |        Number 2.1       |

     |                         |       |                         |

     +-------------------------+       +-------------------------+

     |                         |

     |     Block size=512Kb    |

     |        Number 1.2       |

     |                         |

     +-------------------------+

 

 

A user on a Unix system, uses the standard Unix cat command to concatenate the 2 files into a single larger file. The cat command, simply appends all of the contents of file #2 to file #1 and outputs the concatenated result to file #3.

 

              File #3

     +-------------------------+

     |                         |

     |     Block size=512Kb    |

     |        Number 1.1       |

     |                         |

     +-------------------------+

     |                         |

     |     Block size=512Kb    |

     |        Number 1.2       |

     |                         |

     +-------------------------+

     |                         |

     |     Block size=256Kb    |

     |        Number 2.1       |

     |                         |

     +-------------------------+

 

7.2 A capture file is split into multiple files

 

In section 5.1 we concatenated two files (File #1 and File #2) into a single File #3. Now we will split File #3 into 3 separate files. We will do the split at the Block boundary which is exactly at 512 Kb offsets.

 

Using the Unix command split, we split File #3 into 3 individual files with the following Unix command line:

 

       split -b 512K file3.nap

 

The above command will generate 3 files with names aa, ab, ac each of which is a separate and complete NAP capture file, only containing a single block of data. File aa and ab will have file size of 512 Kb, while file ac will have file size of 256 Kb.

 

The b option, specifies to split the file every given number of bytes, repeatedly, until the end of the file. Last argument is simply the file name to do the split on.

 

8. Data Format

 

All Integer values are stored in big-endian order, with the high-order bits first.

 

 

Security Considerations

 

The NAP file format contain 4 security measures to ensure security and privacy of the data.

 

1)      A simple Security Code set in the Block Record header. Intended to prevent casual user from acquiring Record Data. Access to data is denied at the discretion of the program reading the capture file. Such a program can maliciously ignore the specification outlined in this document and ignore the password set on the Block Record and read the Record Data Field.

2)      A simple hash of all Record fields including Record Data Field. Again this is insecure means of protecting sensitive or private data. The record is hashed using the Security Code in the Block Record Header.

3)      Full encryption using a public and private encryption keys. No passwords or keys are stored, but the data is encrypted using a highly secure algorithm. The user must supply a decryption key in order to read the Record Data Field. The data can be read out in its encrypted form. The record header is not encrypted, but is hashed using security options #2.

4)      Encrypt the entire file using outside program. This method is not described in this memo. This method also disables some of the features of the file format. Externally encrypted files can not be split or concatenated together without first being decoded.

 

In each of the above cases, case #2 and case #3, only the Record Data Field is encrypted, not the Record Header. Also Packet Count field in the Block Record, if present, is readable.

 

The advantages of case #1, case #2 and case #3 is that files can be read out one record at a time. Multiple files can still be concatenated. Capture file can be split into multiple separate files. Also since security measures are set on a Block not entire file, only portions of the file can be encrypted, leaving other blocks unencrypted.
